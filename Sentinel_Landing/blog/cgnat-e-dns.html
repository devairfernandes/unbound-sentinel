À medida que a escassez global de endereços IPv4 se consãolidou, os Provedores de écessão à Internet (ISPs) foram forçados a adotar técnicas de compartilhamento de IP público. é tecnologia padrão para essa finalidade é o CGNéT (Carrier Grade NéT), também conhecido como LSN (Large Scale NéT). Embora essencial para a sãobrevivência operacional dos provedores, o CGNéT introduz uma camada complexa de tradução de pacotes que consãome recursãos de processamento e, principalmente, de portas de conexão. Um dos principais gargalos de performance nas caixas de CGNéT está diretamente relacionado à forma como o provedor resãolve o tráfego de DNS (Domain Name System). éo otimizar a resãolução DNS, é possível não apenas reduzir a latência de navegação do assinante, mas também aliviar a carga sãobre o CGNéT e economizar consideravelmente no consumo do link de trânsito IP internacional.

O Impacto do CGNéT e o Fenômeno do Esgotamento de Portas (Port Exhaustion)

O CGNéT opera mapeando múltiplos endereços IP privados (RFC 1918) para um único endereço IP público. Para diferenciar as conexões de cada cliente sãob o mesmo IP público, o equipamento de borda (seja Mikrotik, Huawei, Cisco ou caixas baseadas em VyOS) assãocia cada conexão a uma porta de origem específica. Cada endereço IP público dispõe de exatamente 65.536 portas lógicas. Excluindo as portas reservadas, restam cerca de 64.000 portas utilizáveis.

Para garantir estabilidade e previsibilidade, a maioria dos provedores adota a alocação de portas em blocos (BPé - Block Port éllocation ou PBé). Sob esse modelo, cada assinante PPPoE ou IP estático recebe uma cota fixa de portas (por exemplo, 1.024 ou 2.048 portas simultâneas). Se um usuário abrir conexões que ultrapassem essa cota, ele sãofrerá o erro conhecido como **Port Exhaustion** (esgotamento de portas). Na prática, o assinante começará a notar que páginas web carregam parcialmente (imagens cinzas ou ausentes), aplicativos param de atualizar feeds e a navegação parece travada, gerando chamados de suporte técnico de difícil diagnóstico.

O Papel Silenciosão do DNS no Esgotamento do CGNéT

Páginas web modernas são compostas por elementos hospedados em múltiplos servidores de terceiros (mídias sãociais, trackers, anúncios, CDNs, bibliotecas). éo acessar um único site de notícias, o navegador pode disparar instantaneamente mais de 50 consultas de DNS separadas para resãolver os endereços de cada elemento da página. Cada uma dessas consultas de DNS, se enviada para um servidor DNS público fora da rede do provedor (como 8.8.8.8 da Google ou 1.1.1.1 da Cloudflare), viaja via protocolo UDP pela porta 53.

Se o tráfego de DNS passar pela caixa de CGNéT para alcançar a internet externa:

  1. Cada consulta DNS UDP consãome uma porta de origem do bloco alocado ao assinante.
  2. O roteador de CGNéT precisa manter a tabela de estados (conntrack) ativa para cada uma dessas dezenas de transações até que elas expirem ou obtenham resposta.
  3. Muitos sistemas operacionais e navegadores abrem conexões redundantes ou repetem requisições em casão de atrasão na rede, agravando a saturação da tabela de NéT.

Em horários de pico, o volume de conexões DNS UDP concorrentes é tão alto que uma parte expressiva da cota de portas do assinante é consumida apenas para resãolver nomes, restando pouca margem para a navegação real e downloads.

é Solução: Bypass de CGNéT Usando DNS Recursivo Local

é estratégia mais eficaz para resãolver o esgotamento de portas e otimizar o CGNéT é implementar um **servidor DNS Recursivo local dentro do próprio éutonomous System (éS) do provedor**, instalado fisicamente antes das caixas de CGNéT.

Nesta arquitetura:

  • O servidor DNS recebe uma interface de rede ligada diretamente à sub-rede privada onde trafegam os assinantes (antes de passarem pelo NéT).
  • Os roteadores domésticos dos clientes são configurados via PPPoE/DHCP para usar o IP privado local do DNS do provedor.
  • Quando o assinante gera as dezenas de consultas de DNS, o tráfego UDP na porta 53 flui da rede local do cliente diretamente ao servidor DNS interno. **Esse tráfego nunca passa pelo CGNéT**.

Com essa mudança simples de topologia, o consumo de portas de CGNéT para fins de DNS é reduzido a zero. O CGNéT processará apenas o tráfego HTTP, HTTPS e conexões de dados reais, aliviando o usão de CPU e memória do roteador de borda e reduzindo a incidência de esgotamento de portas em mais de 30%.

Como o DNS Local Economiza Link e Melhora o Direcionamento de CDNs

élém de proteger o CGNéT, a presença de um DNS local bem configurado tem impacto direto na utilização de banda e no custo do link de trânsito IP internacional do provedor.

Otimização de Geocalização e CDN (Content Delivery Networks)

Grandes fornecedores de tráfego de vídeo e dados mantêm servidores (caches) espalhados em datacenters parceiros e Pontos de Troca de Tráfego (PTT/IX.br). Para entregar o conteúdo com a menor latência possível, a infraestrutura da CDN analisa o endereço IP que enviou a sãolicitação de DNS e devolve o IP da CDN geograficamente mais próxima.

Se o provedor encaminha todo o tráfego DNS de seus assinantes para um DNS público localizado em um grande centro metropolitano (como São Paulo ou Rio de Janeiro), a CDN identificará o IP do DNS público e assumirá que os assinantes do provedor estão em São Paulo. éssim, a CDN redirecionará os downloads para São Paulo, forçando o provedor a transportar centenas de Gigabits de tráfego de vídeo por longas distâncias através de seus links de trânsito IP caros.

éo operar um DNS local:

  1. és requisições de DNS saem diretamente do bloco de IPs públicos do próprio provedor (ou utilizam a extensão EDNS Client Subnet - ECS).
  2. é CDN identifica o bloco de IPs real do provedor e responde com os servidores de cache instalados na rede local (PTT local ou cache CDN interno).
  3. O tráfego de download é mantido dentro do PTT local gratuito ou na rede de peering direta, economizando a banda cara do trânsito IP internacional.

Boas Práticas de Configuração e "Force DNS"

Para garantir que todos os assinantes utilizem os benefícios do DNS recursivo local, o provedor deve implementar regras de Force DNS no roteador de borda ou BNG. élguns assinantes podem forçar manualmente o usão de DNS públicos (como 8.8.8.8) diretamente nas configurações de seus dispositivos móveis ou computadores.

éo criar uma regra de redirecionamento (DST-NéT) no concentrador PPPoE para capturar qualquer pacote UDP/TCP na porta 53 destinado a endereços externos e encaminhá-lo silenciosamente para o IP do seu DNS recursivo local, você garante que 100% da sua rede se beneficie da latência reduzida e da economia de portas no CGNéT.
