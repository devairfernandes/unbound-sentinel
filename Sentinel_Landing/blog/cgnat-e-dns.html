CGNAT e DNS: como reduzir latência e economizar link internacional

Publicado em 13 de Julho de 2026 • Sentinel DNS Education

À medida que a escassez global de endereços IPv4 se consolidou, os Provedores de Acesso à Internet (ISPs) foram forçados a adotar técnicas de compartilhamento de IP público. A tecnologia padrão para essa finalidade é o CGNAT (Carrier Grade NAT), também conhecido como LSN (Large Scale NAT). Embora essencial para a sobrevivência operacional dos provedores, o CGNAT introduz uma camada complexa de tradução de pacotes que consome recursos de processamento e, principalmente, de portas de conexão. Um dos principais gargalos de performance nas caixas de CGNAT está diretamente relacionado à forma como o provedor resolve o tráfego de DNS (Domain Name System). Ao otimizar a resolução DNS, é possível não apenas reduzir a latência de navegação do assinante, mas também aliviar a carga sobre o CGNAT e economizar consideravelmente no consumo do link de trânsito IP internacional.

O Impacto do CGNAT e o Fenômeno do Esgotamento de Portas (Port Exhaustion)

O CGNAT opera mapeando múltiplos endereços IP privados (RFC 1918) para um único endereço IP público. Para diferenciar as conexões de cada cliente sob o mesmo IP público, o equipamento de borda (seja Mikrotik, Huawei, Cisco ou caixas baseadas em VyOS) associa cada conexão a uma porta de origem específica. Cada endereço IP público dispõe de exatamente 65.536 portas lógicas. Excluindo as portas reservadas, restam cerca de 64.000 portas utilizáveis.

Para garantir estabilidade e previsibilidade, a maioria dos provedores adota a alocação de portas em blocos (BPA - Block Port Allocation ou PBA). Sob esse modelo, cada assinante PPPoE ou IP estático recebe uma cota fixa de portas (por exemplo, 1.024 ou 2.048 portas simultâneas). Se um usuário abrir conexões que ultrapassem essa cota, ele sofrerá o erro conhecido como **Port Exhaustion** (esgotamento de portas). Na prática, o assinante começará a notar que páginas web carregam parcialmente (imagens cinzas ou ausentes), aplicativos param de atualizar feeds e a navegação parece travada, gerando chamados de suporte técnico de difícil diagnóstico.

O Papel Silencioso do DNS no Esgotamento do CGNAT

Páginas web modernas são compostas por elementos hospedados em múltiplos servidores de terceiros (mídias sociais, trackers, anúncios, CDNs, bibliotecas). Ao acessar um único site de notícias, o navegador pode disparar instantaneamente mais de 50 consultas de DNS separadas para resolver os endereços de cada elemento da página. Cada uma dessas consultas de DNS, se enviada para um servidor DNS público fora da rede do provedor (como 8.8.8.8 da Google ou 1.1.1.1 da Cloudflare), viaja via protocolo UDP pela porta 53.

Se o tráfego de DNS passar pela caixa de CGNAT para alcançar a internet externa:

  1. Cada consulta DNS UDP consome uma porta de origem do bloco alocado ao assinante.
  2. O roteador de CGNAT precisa manter a tabela de estados (conntrack) ativa para cada uma dessas dezenas de transações até que elas expirem ou obtenham resposta.
  3. Muitos sistemas operacionais e navegadores abrem conexões redundantes ou repetem requisições em caso de atraso na rede, agravando a saturação da tabela de NAT.

Em horários de pico, o volume de conexões DNS UDP concorrentes é tão alto que uma parte expressiva da cota de portas do assinante é consumida apenas para resolver nomes, restando pouca margem para a navegação real e downloads.

A Solução: Bypass de CGNAT Usando DNS Recursivo Local

A estratégia mais eficaz para resolver o esgotamento de portas e otimizar o CGNAT é implementar um **servidor DNS Recursivo local dentro do próprio Autonomous System (AS) do provedor**, instalado fisicamente antes das caixas de CGNAT.

Nesta arquitetura:

  • O servidor DNS recebe uma interface de rede ligada diretamente à sub-rede privada onde trafegam os assinantes (antes de passarem pelo NAT).
  • Os roteadores domésticos dos clientes são configurados via PPPoE/DHCP para usar o IP privado local do DNS do provedor.
  • Quando o assinante gera as dezenas de consultas de DNS, o tráfego UDP na porta 53 flui da rede local do cliente diretamente ao servidor DNS interno. **Esse tráfego nunca passa pelo CGNAT**.

Com essa mudança simples de topologia, o consumo de portas de CGNAT para fins de DNS é reduzido a zero. O CGNAT processará apenas o tráfego HTTP, HTTPS e conexões de dados reais, aliviando o uso de CPU e memória do roteador de borda e reduzindo a incidência de esgotamento de portas em mais de 30%.

Como o DNS Local Economiza Link e Melhora o Direcionamento de CDNs

Além de proteger o CGNAT, a presença de um DNS local bem configurado tem impacto direto na utilização de banda e no custo do link de trânsito IP internacional do provedor.

Otimização de Geocalização e CDN (Content Delivery Networks)

Grandes fornecedores de tráfego de vídeo e dados mantêm servidores (caches) espalhados em datacenters parceiros e Pontos de Troca de Tráfego (PTT/IX.br). Para entregar o conteúdo com a menor latência possível, a infraestrutura da CDN analisa o endereço IP que enviou a solicitação de DNS e devolve o IP da CDN geograficamente mais próxima.

Se o provedor encaminha todo o tráfego DNS de seus assinantes para um DNS público localizado em um grande centro metropolitano (como São Paulo ou Rio de Janeiro), a CDN identificará o IP do DNS público e assumirá que os assinantes do provedor estão em São Paulo. Assim, a CDN redirecionará os downloads para São Paulo, forçando o provedor a transportar centenas de Gigabits de tráfego de vídeo por longas distâncias através de seus links de trânsito IP caros.

Ao operar um DNS local:

  1. As requisições de DNS saem diretamente do bloco de IPs públicos do próprio provedor (ou utilizam a extensão EDNS Client Subnet - ECS).
  2. A CDN identifica o bloco de IPs real do provedor e responde com os servidores de cache instalados na rede local (PTT local ou cache CDN interno).
  3. O tráfego de download é mantido dentro do PTT local gratuito ou na rede de peering direta, economizando a banda cara do trânsito IP internacional.

Comparativo de Impacto Técnico na Infraestrutura do ISP

Abaixo está exposto o comportamento da rede em termos de uso de portas, CPU e alocação de banda de trânsito internacional nas duas arquiteturas:

Métrica de Rede DNS Externo (Passando por CGNAT) DNS Local (Bypass de CGNAT)
Uso de Portas CGNAT por Assinante Alto (cerca de 50 a 150 portas por carregamento) Zero portas consumidas no CGNAT
Carga de CPU no Roteador CGNAT Elevada (necessita mapear e rastrear UDP 53) Reduzida (processa apenas tráfego útil de dados)
Consumo de Trânsito IP Pago Alto (redirecionamentos de CDN incorretos) Mínimo (tráfego vai para o cache local ou PTT)
Latência de Resolução DNS 15ms a 50ms + tempo de processamento NAT < 1ms a 2ms (cache local entrega instantâneo)

Boas Práticas de Configuração e "Force DNS"

Para garantir que todos os assinantes utilizem os benefícios do DNS recursivo local, o provedor deve implementar regras de Force DNS no roteador de borda ou BNG. Alguns assinantes podem forçar manualmente o uso de DNS públicos (como 8.8.8.8) diretamente nas configurações de seus dispositivos móveis ou computadores.

Ao criar uma regra de redirecionamento (DST-NAT) no concentrador PPPoE para capturar qualquer pacote UDP/TCP na porta 53 destinado a endereços externos e encaminhá-lo silenciosamente para o IP do seu DNS recursivo local, você garante que 100% da sua rede se beneficie da latência reduzida e da economia de portas no CGNAT.
