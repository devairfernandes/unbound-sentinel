Documentação Oficial Sentinel DNS

Abaixo você encontra a referência técnica completa de arquitetura, instalação, configuração, hardening e troubleshooting do appliance Sentinel DNS (versões FREE, PRO Elite e PRO Lite).

1. Visão Geral

O Sentinel DNS é um appliance (distribuído via ISO Rocky Linux) que entrega um servidor DNS recursivo Hyperlocal de alta performance, fundindo o motor Unbound com uma camada integrada de Cyber Threat Intelligence (CTI). Seu foco é resolver DNS com resiliência, atuar como firewall DNS contra ameaças e botnets, e permitir gestão local ou remota.

  • Unbound Automático: Responsável pela resolução recursiva.
  • Camada de Segurança CTI: Motor inteligente que avalia requisições em tempo real e bloqueia malware, ads e ameaças baseadas em feeds de reputação (OSINT).
  • Painel Node.js Local: Servidor web Express embarcado (Central de Configurações) que manipula configurações base (via SSH local/Child Process) e exibe estatísticas em tempo real sem expor serviços do SO.
  • Sincronização Master: Comunicação criptografada (Token HMAC) com o Master Server para validação de licenças, sincronismo HA, download de bases como GeoIP e recebimento de Threat Intel dinâmico.

2. Instalação

O Sentinel é projetado para instalação unattended (autônoma) usando Kickstart. Ele prepara o sistema Minimal, desativa serviços desnecessários e sobe a stack do Unbound + Node.js.

Requisitos Mínimos (Dimensionamento)

Passo a Passo via ISO (Auto-deployable)

  1. Faça boot pela ISO em sua máquina virtual ou hardware. O processo de Kickstart (ks.cfg) roda 100% offline, zerando a necessidade de repositórios externos.
  2. No primeiro boot, o sistema apresentará um assistente em modo texto para configuração de rede (firstboot-network.sh) solicitando IP, Máscara e Gateway.
  3. Após confirmar a rede, o painel local Node.js ficará acessível via HTTP no IP configurado. Acesse pelo navegador web.
  4. No primeiro acesso, você deve ativá-lo inserindo sua Chave de Licença. Esse processo atrela o servidor (via HWID) ao Master Server, destravando o painel e habilitando o download de assinaturas, GeoIP e CTI do plano correspondente.

3. Central de Configurações (Painel Administrativo)

A aba Configurações do Sistema permite o controle granular do host Rocky Linux e do daemon do Unbound diretamente pela interface, eliminando comandos via SSH para operações de rotina.

Unbound DNS (Avançado)

Interface direta para manipular arquivos `.conf` customizados que rodam junto com a instância local. Permite ajustes de performance adicionais e tuning sem precisar editar /etc/unbound/unbound.conf manualmente no bash. Sempre execute o Checkconf na tela antes de aplicar.

Controle de Acesso (IPs)

Gerenciamento do access-control do Unbound. Limita quais redes ou IPs podem disparar queries contra este servidor recursivo (evitando se tornar um Open Resolver). Veja também a seção Segurança CTI para controle avançado e bloqueios.

Firewall (Security)

Gerenciamento de ACLs e regras a nível de SO (Firewalld). O módulo injeta regras nativas de filtragem e mapeamento de portas sem exigir domínio de firewall-cmd, fechando a máquina contra ataques diretos ou limitando portas (como SSH 22, HTTP 80/3000, DNS 53).

Rede (Interfaces)

Edita as configurações físicas e virtuais da máquina (NetworkManager). Permite trocar IPv4, máscara de sub-rede, habilitar IPv6 e gerenciar roteamento estático (Gateway) mantendo a estabilidade da interface.

Layout do Painel

Customização visual. Habilita ou desabilita a exibição de Cards de Telemetria Específicos e Gráficos no Dashboard principal, reduzindo a carga visual dependendo do operador logado.

Sistemas Internos (Static)

Cadastro de zonas de resolução offline local. Adiciona entradas no Unbound para responder resoluções sem acessar a raiz da internet (ideal para gerenciar redes intranet, painéis corporativos ou subdomínios locais no formato nome.empresa.local).

Blacklist (Domínios)

Módulo simplificado para bloqueios rápidos. Interliga-se com a seção CTI, enviando domínios para a zona de rejeição instantânea local-zone, devolvendo resposta NXDOMAIN imediata a clientes que buscarem os itens ali adicionados.

Credenciais & SSH

Alteração das senhas padrão do sistema e administração do serviço SSH. O painel altera a senha do usuário administrativo do próprio host Linux no backend.

GeoIP / MaxMind

Gestão do banco de geolocalização IP que alimenta os gráficos globais de tráfego.

  • Prioridade de Detecção: O servidor resolve a localização usando primeiramente o Banco Local MaxMind. Se o IP não constar, faz fallback para a Web API da MaxMind, e por último aciona o provedor gratuito genérico (ip-api.com).
  • Provedores: O ip-api.com possui limite estrito de 1.000 requisições/dia. Já o banco local atua com queries ilimitadas sem latência.
  • Banco Local MaxMind: O painel permite baixar o arquivo GeoLite2-City.mmdb diretamente do Master Server com 1 clique (para planos PRO) ou importar de arquivo via upload, sendo salvo em /opt/unbound-dashboard/GeoLite2-City.mmdb. A opção de download pelo Master elimina a necessidade do administrador manter uma conta MaxMind própria e renovar a License Key periodicamente, unificando a distribuição de bases.
  • Use o botão Testar Agora para forçar a resolução de um IP alvo.

Data, Hora & Timezone

Sincroniza o relógio do sistema com NTP servers. Essencial para validação de certificados DNSSEC, logs de auditoria CTI e geração do Token HMAC. Um timezone desajustado impedirá a verificação de assinaturas digitais nas respostas DNS.

Geoblocking por País

Bloqueio massivo e determinístico a nível de kernel.

  • Funcionamento: Nenhum país vem bloqueado por padrão. Quando um país é adicionado, o sistema cria dinamicamente listas ipset cruzadas com iptables utilizando as bases do ipdeny.com.
  • Sintaxe: Exige o uso do padrão ISO 3166-1 alpha-2 com duas letras maiúsculas (ex: CN para China, RU para Rússia, KP para Coreia do Norte).
  • Pacotes vindos de ASN/Ranges desses países são barrados silenciosamente na borda do Linux, poupando CPU e a pilha de rede do Unbound.

Integração AnaBlock

Módulo vitalício focado em conformidade regulatória para provedores de internet (ISPs) no Brasil (ANATEL).

  • Como Funciona: Quando habilitado, o Sentinel consome a API Judicial da AnaBlock (verificando os endpoints domains/all, url/all, ipv4/block e ipv6/block).
  • Segurança & Validação: O serviço valida a resposta da API garantindo que as diretivas local-zone retornadas possuem sintaxe estrita, gravando-as no arquivo local e forçando o reinício do cache DNS.
  • Frequência: A atualização ocorre de forma silenciosa e periódica. Se houver timeout ou erro 401/403 (falta de autorização IP no portal AnaBlock), o sistema notifica nos logs do backend e preserva o cache pré-existente (sem causar downtime).
  • Nota: Esse recurso deve ser autorizado por IP previamente junto à agência antes de o sync funcionar sem erro 403.

4. Segurança CTI (Cyber Threat Intelligence)

Módulo Central: Segurança CTI

Os domínios categorizados como maliciosos recebem Sinkhole (resposta NXDOMAIN ou Redirecionamento para a Página de Bloqueio) em sub-milissegundos via backend. O painel centraliza a gestão de segurança em 7 sub-módulos:

1. Log de Consultas (Tempo Real)

Exibe o tráfego de resolução DNS instantâneo do servidor. É possível filtrar as requisições por Domínio ou IP do Cliente. Cada registro exibe o Horário, Cliente, Domínio, Tipo de registro e Status da requisição. Possui atalhos de ação rápida para enviar qualquer domínio investigado direto para a Blacklist ou Whitelist com um clique.

2. OSINT e Ameaças

Painel de interceptações de ameaças categorizadas com retenção em tela de 2 horas. O sistema permite auditar domínios bloqueados agrupados por categorias.

3. Gravity (Adlists)

Gerenciador de listas de bloqueio baseadas em hosts ou domínios em massa (Adlists). O administrador pode adicionar URLs para arquivos `.txt` contendo malwares ou anúncios, e utilizar o botão Sincronizar (Gravity) para processar, deduplicar e injetar os bloqueios no motor do Unbound de forma compilada, preservando a performance da memória RAM.

4. Lista Branca (Whitelist) & Lista Negra (Blacklist)

Módulo de Override local manual. Permite cadastrar exceções forçadas. O sistema aceita domínios absolutos (ex: malware.net) e curingas/wildcards (ex: *.malware.net) nativamente, sobrepondo qualquer regra do CTI ou Gravity.

5. Bloqueio de Apps & Controle Parental

Controle semântico de tráfego, focado em redes empresariais e provedores de internet:

  • Perfis de Filtro & IPs Alvo: As regras de bloqueio podem ser aplicadas globalmente (afetando todos os clientes do DNS) ou segmentadas por IPs e Sub-redes (ex: 192.168.1.0/24), permitindo a criação de múltiplos perfis de filtragem no mesmo servidor.
  • Busca Segura Forçada (SafeSearch): Ao habilitar esta chave, o Unbound forçará ativamente o modo restrito nos motores do Google, YouTube, Bing e DuckDuckGo reescrevendo as requisições CNAME internamente.
  • Página de Bloqueio Customizada: Em vez de gerar "NXDOMAIN" para sites proibidos, redireciona o tráfego web do usuário final para uma landing page local do appliance explicando o motivo do bloqueio (exige que o cliente não utilize HSTS/HTTPS estrito para ser exibida).
  • Bloqueio de Serviços: Chaves de 1 clique para derrubar acesso a Sites Adultos, TikTok, YouTube, Facebook, Instagram, Netflix, Roblox, Tinder. Administradores podem cadastrar "Novos Serviços" adicionando regex ou strings que identificam os ASNs ou domínios da aplicação.

6. Zonas e Apontamentos DNS (Local-Data)

Substitui a necessidade de manter um servidor Bind9 apenas para zonas locais. Permite criar registros autoritativos para a rede interna (ex: A, CNAME, TXT) resolvendo instantaneamente. Útil para sistemas internos corporativos e intranets.

7. Sincronismo HA (High Availability)

Módulo para redes com múltiplos appliances Sentinel em balanceamento. A configuração suporta o modelo Ativo-Ativo lógico através do espelhamento de regras:

  • Master: O nó eleito gerará um "Token de Sincronismo" e enviará as configurações locais.
  • Replica: O nó secundário recebe o IP do Master e o Token (conectando-se à porta API 3000), sincronizando as Blacklists, Filtros de Apps, Gravity e Zonas DNS em background periodicamente, dispensando dupla configuração manual.

Indicadores do Dashboard CTI

O topo da interface fornece a telemetria ao vivo da saúde da rede baseada nas categorizações do OSINT:

  • Ameaças Críticas: Contador em tempo real (últimas 2h) de requisições barradas correspondentes a bases de OSINT (Severidade: CRÍTICO).
  • Atividade Suspeita: Requisições barradas por baixa reputação (Severidade: SUSPEITO).
  • IPs Monitorados: Total de IPs únicos ativos gerando tráfego contra o servidor.
  • Bloqueios (Blacklist): Volume de respostas suprimidas pelas regras de Lista Negra manuais ou Adlists locais.

5. Ping Master (Monitor de Latência)

O Ping Master é um sistema ativo embarcado (NOC Telemetry Elite) para monitorar e auditar a conectividade da rede a partir da visão do servidor DNS. Ele dispara testes em background de forma contínua para alvos cruciais definidos pelo administrador.

Métricas Consolidadas (Cards)

  • Média Geral (ms): Avalia a latência média de todos os hosts com status online.
  • Serviços Ativos: Contagem de alvos respondendo com sucesso ao monitoramento.
  • Latência Alta: Contagem de hosts ultrapassando o threshold estrito de 150ms.
  • Serviços Offline: Hosts cujo teste reportou timeout (falha total ou pacotes perdidos em sucessão).

Monitoramento Individual de Alvos

Na visualização detalhada, a grade "Latências de Conectividade" exibe cada servidor monitorado acompanhado de um Sparkline em tempo real (gráfico em onda visual). Cada card consolida:

  • Ping Exato: Retorna o tempo de ida e volta (RTT) em milissegundos.
  • Jitter (Variação): Baseado no desvio padrão entre as sucessivas medições válidas. Útil para prever engasgos em VoIP ou streaming.
  • % Perda (Packet Loss): Razão entre falhas e sucesso sobre o histórico mais recente de testes.

Escala e Thresholds de Qualidade

O painel categoriza a saúde dos serviços automaticamente com as seguintes réguas reais avaliadas via código (cores verde, laranja, vermelho):

  • Excelente: Menor que 80ms (Status good).
  • Instável (Warning): Entre 80ms e 159ms (Status warning). Alerta o NOC para possível congestionamento de rota.
  • Latência Alta (Crítico): 160ms ou maior (Status bad). Degradação severa.
  • Fora do Ar (Offline): Teste falhou por timeout contínuo ou ICMP unreachable.

Cadastrando um Novo Alvo

Clique em + Novo Alvo. O formulário exige:

  1. Nome do Serviço: Identificador (Ex: DNS Google, Link Dedicado).
  2. Endereço IP ou Domínio: O destino alvo para envio de pacotes (Ex: 8.8.8.8, netflix.com).
  3. Método (Tipo de Alvo): Opções como ICMP (Ping Tradicional) ou Check de Porta TCP/UDP (testando 3-way handshake em porta customizada, bom para evitar falsos positivos de firewalls que descartam ICMP).

7. Troubleshooting Comum & Sysadmin

Ao realizar depuração no terminal Linux local:

Logs em Tempo Real (Journald)

# Monitorar o tráfego e logs de erro do motor Unbound
journalctl -u unbound -f

Validação de Configuração (Checkconf)

Se você alterar regras diretamente via bash em /etc/unbound/ em vez de usar a tela Configurações Avançadas, sempre valide a sintaxe:

unbound-checkconf
# Se modificar access-control manualmente via terminal, a linha "server:" deve iniciar o bloco!

Reprocessar Rede (Modo Emergência)

Para religar o prompt interativo em casos de falha crônica de rede ou mudança de gateway local:

rm -f /root/.firstboot_done && /root/firstboot-network.sh

Teste Externo de Open Resolver

Por padrão, a recursão DNS só permite tráfego de IPs autorizados em Controle de Acesso. Testar respostas não autorizadas gerará falhas de pacote ou alertas visuais para prevenir envolvimento em ataques DDoS Amplification.

8. FAQ Técnico

  • O Unbound perde cache ao reiniciar (zerando hits)?
    Não. O Sentinel utiliza Persistent Cache através do unbound-control load_cache invocado no boot via systemd. Ele grava dump na memória antes de desligar e recarrega em milissegundos sem esbarrar no disco, evitando lentidão matinal na rede de ISPs.
  • Por que algumas atualizações Gravity param num plano Vitalício?
    Plataformas PRO Lite possuem a licença de uso do DNS para sempre (nunca expiram), mas o consumo e curadoria de bandas pesadas de feeds de ameaças, malwares frescos e a base MaxMind GeoLite2 exigem custo contínuo de nuvem e inteligência. É preciso manter o Add-on CTI para que as queries à API não retornem limites.
  • Posso criar regras de IPtables brutas?
    Não é recomendado contornar o firewalld no Rocky Linux do Sentinel. Use o módulo de Firewall embutido. Ao bloquear tráfego DNS para ranges específicos pelo Firewalld, o motor subjacente desvincula o serviço da zona automaticamente na camada kernel de forma limpa.
